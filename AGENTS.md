# Momogo 서버 에이전트 지침

## 프로젝트 환경

- Kotlin 2.3, Java 21, Spring Boot 4.1과 Gradle Wrapper를 사용한다.
- 이 프로젝트는 Spring MVC 애플리케이션이다. 코드에서 명시적으로 도입하지 않은 한 리액티브 또는 코루틴 동작을 가정하지 않는다.
- 생성자 주입을 사용하고 의존성을 명시적으로 드러낸다.
- 전체 검증은 `./gradlew check`로 실행한다. 테스트만 다시 실행할 때는 `./gradlew test`를 사용한다.

## Git 작업 규칙

### 브랜치

- 기능 개발 브랜치는 `feature/{작업-내용}` 형식으로 생성한다.
- 설정, 빌드, 문서, 의존성 등 기능 외 작업 브랜치는 `chore/{작업-내용}` 형식으로 생성한다.
- `{작업-내용}`은 영문 소문자와 하이픈(`-`)을 사용해 짧고 구체적으로 작성한다.
- 예: `feature/global-exception-handler`, `chore/github-actions-ci`

### 커밋

- 기능 개발 커밋은 `feature: {변경 내용}` 형식으로 작성한다.
- 기능 외 작업 커밋은 `chore: {변경 내용}` 형식으로 작성한다.
- 제목은 한글로 간결하게 작성하고 무엇을 변경했는지 명확하게 드러낸다.
- 한 커밋에는 하나의 논리적인 변경만 포함한다.
- 예: `feature: 전역 예외 처리 추가`, `chore: GitHub Actions CI 구성`

### Pull Request

- PR 제목은 관련 커밋과 같은 접두사를 사용한다. 예: `feature: 전역 예외 처리 추가`
- PR 본문은 IT 분야에서 일하는 고등학생이 추가 설명 없이 이해할 수 있는 수준의 쉬운 한글로 작성한다.
- 어려운 용어와 긴 배경 설명을 피하고, 꼭 필요한 경우 용어의 뜻을 한 문장으로 설명한다.
- 변경 이유와 핵심 변경 사항만 짧게 작성한다. 각 항목은 1~2문장으로 제한한다.
- 테스트를 실행했다면 사용한 명령과 결과를 한 줄로 작성한다.
- 권장 형식:

  ```markdown
  ## 왜 바꿨나요?
  {변경 이유를 짧게 설명}

  ## 무엇을 바꿨나요?
  - {핵심 변경 사항}

  ## 확인
  - `./gradlew check` 통과
  ```

## 코드 리뷰 기준

동작, 정확성, 보안, 회귀 위험을 중심으로 검토한다. 구체적인 결함을 유발하지 않는 포맷, 명명 취향, 추측성 미래 문제는 지적하지 않는다.

### 리뷰 언어

- 리뷰 제목, 본문, 요약과 수정 제안은 모두 한글로 작성한다.
- 클래스명, 함수명, 변수명, API 경로, 예외명과 코드 조각은 원문을 유지한다.
- 필요한 기술 용어는 한글 설명 뒤에 영문을 괄호로 함께 표기할 수 있다.
- 간결하고 직접적으로 작성하며, 번역투나 불필요한 수식어를 피한다.

### 심각도

- P0: 보안 침해, 데이터 손실·훼손, 장시간의 운영 장애 또는 비밀정보·개인정보의 광범위한 노출을 일으킬 수 있는 문제다.
- P1: 외부에서 관찰되는 잘못된 동작, 지원 중인 API 계약 위반, 필수 검증·인가 우회, 현실적인 런타임 실패 또는 의미 있는 회귀를 일으킬 수 있는 문제다.
- 사소한 유지보수성, 스타일, 문서 또는 선택적인 테스트 개선 사항을 P0/P1으로 올리지 않는다.

### API 계약과 입력값 검증

- HTTP 메서드, 경로, 상태 코드, 헤더와 응답 본문이 의도한 API 계약을 유지하는지 확인한다.
- 버전 변경이나 클라이언트와의 조율 없이 요청·응답 규격을 호환되지 않게 변경한 경우 지적한다.
- 신뢰할 수 없는 요청값이 시스템 경계에서 검증되는지 확인한다. 중첩된 요청 객체에도 검증이 적용되는지, Kotlin 애노테이션의 사용 지점 대상이 의도한 필드나 파라미터인지 확인한다.
- 값의 누락, `null`, 공백, 빈 값, 0과 기본값을 의도적으로 구분한다. 요청, 설정 또는 외부 시스템의 영향을 받는 값에 `!!`이나 안전하지 않은 캐스팅을 사용하지 않는다.
- 검증 실패와 도메인 오류가 안정적이고 적절한 HTTP 응답으로 변환되는지 확인한다. 프레임워크 예외가 노출되거나 의도하지 않은 500 응답이 발생하지 않아야 한다.

### Spring 동작

- 컴포넌트 탐색, Bean 조건, 프로필과 설정 바인딩 변경이 애플리케이션 시작 실패나 환경별 오동작을 일으키지 않는지 확인한다.
- 트랜잭션, 비동기, 캐시, 검증, 보안 애노테이션을 Spring 프록시 경계 관점에서 확인한다. 자기 호출이나 가시성·final 문제로 애노테이션이 적용되지 않으면 지적한다.
- 컨트롤러는 전송 계층의 책임에 집중한다. 다른 진입점에서 우회될 수 있는 비즈니스 규칙이 컨트롤러에만 구현되어 있으면 지적한다.
- 싱글턴 Bean이 요청별 가변 상태를 보관하지 않는지, 공유 가변 상태가 스레드 안전한지 확인한다.
- Actuator 노출 범위, 오류 상세 정보, CORS, 인증, 인가와 보안 헤더 변경은 보안에 민감한 변경으로 취급한다.

### Kotlin 정확성

- 명시적으로 `null`을 처리하고, 닫힌 도메인 상태에 대해서는 빠짐없는 `when` 표현식을 사용한다.
- 동등성 비교, 복사와 컬렉션 연산에서 의도하지 않은 변경이나 순서·고유성 손실이 없는지 확인한다.
- 호출자가 구분해야 하는 실패를 숨기는 광범위한 예외 처리, `runCatching` 또는 nullable 대체값을 지적한다.
- 관련 타입이 도입되면 경계값 변환에서 오버플로, 정밀도 손실, 시간대 오류, 로케일 의존성과 잘못된 enum·식별자 파싱을 확인한다.
- Kotlin의 null 안정성을 무력화하거나 프레임워크 경계에서 플랫폼 타입을 모호하게 만드는 Java식 패턴을 피한다.

### 오류, 로깅과 설정

- 스택 트레이스, 내부 예외 메시지, 비밀값, 토큰, 인증정보와 개인정보를 API 응답이나 로그에 노출하지 않는다.
- 예외를 변환할 때 원인을 보존하되, 클라이언트에는 안전한 정보만 반환한다.
- 비밀정보를 저장소에 커밋하지 않는다. 운영 환경에 민감한 값은 환경변수나 비밀 관리 시스템에서 가져오며, 안전하지 않은 기본값이 `prod`에 조용히 적용되지 않아야 한다.
- 프로필별 설정이 공통 설정을 의도치 않게 덮어쓰지 않는지 확인한다. 설정 동작이 중요한 테스트에서는 `test` 프로필을 사용한다.

### 테스트

- 동작 변경에는 성공 경로와 가장 중요한 실패 또는 경계 경로를 검증하는 테스트를 포함한다.
- 컨트롤러 테스트에서는 상태 코드, 콘텐츠 타입과 응답 본문을 검증한다. 변경의 영향을 받는 경우 잘못된 입력과 예외 변환도 검증한다.
- 동작을 증명할 수 있는 가장 좁은 범위의 Spring 테스트를 선호한다. 애플리케이션 배선이나 Spring 경계 전체가 검증 대상일 때만 `@SpringBootTest`를 사용한다.
- 변경된 동작의 회귀 가능성이 크고 결함의 영향이 중대한 경우에만 테스트 누락을 P1으로 판단한다. 테스트되지 않은 모든 줄을 지적하지 않는다.
- 테스트 통과는 보조 증거일 뿐 구현의 정확성을 보장하지 않는다. 단언문이 실제 변경 동작을 검증하는지 확인한다.

### 리뷰 댓글 작성법

- 가장 작은 관련 줄 범위를 지정하고, 구체적인 실패 상황, 영향을 받는 호출자나 환경, 기존 검사가 문제를 막지 못하는 이유를 한글로 설명한다.
- 해당 PR이 새로 만들거나 노출한 문제임을 확인하지 못하면 지적하지 않는다.
- 발견한 각 문제에는 재현 조건이나 실제 실행 흐름을 포함한다.
- P0 또는 P1 문제가 없다면 가치가 낮은 지적을 억지로 만들지 않고 승인한다.
